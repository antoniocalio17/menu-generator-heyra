================================================================================
HEYRA — MENU GENERATOR  |  Architecture & Developer Notes
================================================================================

OVERVIEW
--------
Generates a weekly canteen lunch menu for two tracks (meat / vegetarian),
Monday to Friday, grounded in a real product catalogue (products.csv).

Pipeline:
  Catalogue → Composer → LLM (naming only) → fix_budget → Validator → Exporter → API → UI

The LLM does NOT select ingredients. The composer algorithm builds dishes
from the catalogue using cuisine-weighted sampling. The LLM only names each
dish, writes a recipe overview, and flags genuinely incoherent combinations.

Run with:
  uvicorn api.main:app --reload    (from the menu-generator/ directory)
  Then open http://localhost:8000


================================================================================
MODULE: engine/constants.py
================================================================================

PURPOSE
  Single source of truth for all shared constants. Import from here — never
  redefine in individual modules.

CONSTANTS
  DAYS               — ["monday", ..., "friday"]
  VEGETARIAN_CLASSES — frozenset {"vegan", "vegetarian"}
  PROTEIN_ROLE       — {"meat": "protein_meat", "vegetarian": "protein_veg"}
  DISH_ROLES         — ["carb", "vegetable", "sauce"]
  ROLE_GROUPS        — maps 5 roles to their ingredient groups (see below)
  MEAT_GROUPS        — frozenset of protein_meat ingredient groups
  WEEKLY_BUDGET_EUR  — 20.10
  CUISINE_ROTATION   — ["mediterranean", "asian", "nordic",
                         "middle_eastern", "mediterranean"]
                       Five cuisine targets for Mon–Fri (Med appears twice —
                       richest product pool).

ROLE_GROUPS
  protein_meat  — meat, poultry, fish, seafood, charcuterie
  protein_veg   — legumes, eggs, plant proteins, cheeses, nuts
  carb          — grains, rices, pastas, breads
  vegetable     — vegetables, mushrooms
  sauce         — sauces, condiments, oils, herbs, spices,
                  creams and butters, yogurts

  NOTE: fruits, sweeteners, seeds, flours, milks, plant-based beverages are
  intentionally excluded from every role. They exist in the catalogue and
  chefs can add them manually via Edit. Auto-composition avoids them because
  they produce incoherent sauce slots (e.g. blueberries as a sauce).

MODULE: engine/catalogue.py
================================================================================

PURPOSE
  Loads products.csv and provides typed, filtered samples of products
  by recipe role and dietary track.

DATA TYPES
  Product (TypedDict)
    One row from products.csv:
      product_id, product_name, ingredient_group, cuisine_tag,
      cost_per_100g_eur, energy_kcal_per_100g, dietary_class,
      allergen_gluten, allergen_nuts, allergen_dairy.
    cuisine_tag added by eda.py — one of:
      mediterranean, asian, nordic, middle_eastern, universal.

  IngredientInput (TypedDict)   — (product_id, quantity_g) pair
  MealStats (frozen dataclass)  — cost_eur, calories_kcal

CLASS: Catalogue
  __init__(csv_path)
    Reads CSV, keeps only is_available=1 rows.
    _df     — full filtered DataFrame for sampling
    _by_id  — indexed by product_id for O(1) lookup

  get_candidates_by_role(role, track, n=15, exclude_ids=None)
    Returns up to n randomly sampled Products for a role+track.
    Vegetarian: filters dietary_class ∈ {vegan, vegetarian}
    Meat + protein_meat: filters dietary_class == "meat"

  build_weekly_pools(n_per_role=15, exclude_ids=None)
    Legacy helper (kept for tests). Builds a pool dict across all roles.
    Not used in the main pipeline — composer calls get_candidates_by_role
    directly per dish slot.

  get_product_by_id(product_id)   — O(1) lookup, returns Product or None
  get_cheaper_in_group(group, max_cost, track)  — used by fix_budget
  compute_meal_stats(ingredients)               — returns MealStats
  get_all_products()                            — returns full catalogue list


================================================================================
MODULE: engine/composer.py
================================================================================

PURPOSE
  Builds 10 dish skeletons (5 meat + 5 vegetarian) by weighted sampling
  from the catalogue. No LLM involved — purely algorithmic.

HOW IT WORKS
  For each dish slot, one product is selected per role:
    protein (meat or veg) + carb + vegetable + sauce

  Each slot has a target cuisine (from CUISINE_ROTATION). Products are
  sampled with weights:
    same cuisine as target  →  weight 3
    universal               →  weight 2
    other cuisine           →  weight 1

  This boosts cuisine coherence while allowing cross-cuisine dishes.

DEFAULT QUANTITIES
  protein_meat / protein_veg  150g
  carb                        120g
  vegetable                   100g
  sauce                        30g

UNIQUENESS GUARANTEE
  Each dish's ingredient set (frozenset of product_ids) is tracked.
  _compose_dish() retries up to 20 times per slot to avoid duplicates.

KEY TYPES
  ComposedDish (dataclass)
    cuisine: str
    products: list[tuple[Product, float]]   (product, quantity_g)
    .ingredient_set() → frozenset[int]
    .to_dish(name, description) → Dish

KEY FUNCTIONS
  compose_week(track, catalogue) → list[ComposedDish]   (5 dishes, Mon–Fri)
  compose_single_dish(track, cuisine, catalogue, seen_sets)
    Public wrapper for re-composing one slot (used in groq_llama.py when
    the LLM flags a dish as incoherent).


================================================================================
MODULE: engine/groq_llama.py
================================================================================

PURPOSE
  Calls the Groq LLM (llama-3.3-70b-versatile via OpenAI-compatible API)
  to name and describe pre-composed dishes, and to flag incoherent ones.

WHAT THE LLM DOES
  Receives: 10 composed dishes (4 ingredients each, with product IDs)
  Returns per dish:
    dish_name    — creative name (3–6 words)
    description  — 1–2 sentence recipe overview (technique + assembly,
                   no experience sentences, no generic fluff)
    ingredients  — list of product_ids (from the ones provided; may drop one)
    valid        — false ONLY if combination is genuinely inedible/absurd
                   (e.g. candy or ice cream in a savoury main)
    reason       — brief explanation when valid=false

WHAT THE LLM DOES NOT DO
  - Select ingredients (composer handles this)
  - Access the full catalogue
  - Invent new product IDs

CONFIGURATION
  Model:        llama-3.3-70b-versatile
  Temperature:  0.4
  Response:     json_object mode (json_schema not supported by this model)
  Retries:      3 attempts max
  Backoff:      [2s, 5s, 10s] on Groq 429 rate-limit errors

RETRY LOGIC
  1. JSON parse / Pydantic validation failure  → error fed back, retry
  2. Hallucinated product ID                   → _validate_ids() catches it,
                                                  error fed back, retry
  3. valid=false dishes                        → re-compose those slots via
                                                  compose_single_dish(), retry
     On last attempt: accept anyway + log warning (chef can fix via Edit)
  4. Rate limit (429)                          → backoff, does not count
                                                  against _MAX_RETRIES
  5. Any other API error                       → raises PlannerError immediately

PYDANTIC MODELS
  NamedDish       — dish_name, description, ingredients, valid, reason
  NamingResponse  — meat: list[NamedDish] (5), vegetarian: list[NamedDish] (5)

KEY FUNCTIONS
  generate(week_start, catalogue) → WeeklyPlan
    Full pipeline: compose → prompt → parse → validate → assemble
  _format_dish(dish, idx)         — formats one ComposedDish for the prompt
  _prompt(meat, veg, errors)      — builds system + user messages
  _validate_ids(naming, ...)      — checks all returned IDs are from composed set
  _assemble_plan(...)             — calls dish.to_dish() and builds WeeklyPlan


================================================================================
MODULE: engine/output_format.py
================================================================================

PURPOSE
  Pydantic models that define the stored weekly menu format.

MODELS
  Ingredient   — product_id (int), quantity_g (float > 0)
  Dish         — dish_name (non-empty str), description (str, default ""),
                 ingredients (non-empty list[Ingredient])
  TrackPlan    — one Dish per day: monday … friday
  WeeklyPlan   — week_start (date), meat (TrackPlan), vegetarian (TrackPlan)

  Menus are saved to data/menus/{year}_w{week:02d}.json using WeeklyPlan.


================================================================================
MODULE: engine/validator.py
================================================================================

PURPOSE
  Budget auto-fix and hard constraint validation.

fix_budget(plan, catalogue) → WeeklyPlan
  Called before validate(). Silently adjusts if over budget.
  Strategy: find the single most expensive ingredient, swap it for the
  cheapest available product in the same ingredient_group, skip swaps
  that would duplicate another dish's ingredient set. Repeats until
  within budget or no cheaper swap is possible.

validate(plan, catalogue)
  Collects ALL violations (does not fail fast), raises
  PlanValidationError(errors: list[str]) if any found.
  Checks:
    - All product_ids exist in catalogue
    - Meat track: every dish has ≥1 meat/poultry/fish/seafood/charcuterie
    - Vegetarian track: no ingredient with dietary_class == "meat"
    - Each track weekly cost ≤ WEEKLY_BUDGET_EUR (20.10 EUR)

MODULE: engine/exporter.py
================================================================================

PURPOSE
  Converts a validated WeeklyPlan into a rich WeekSummary with cost,
  calories, allergens, and description. Renders to JSON (API) or Markdown.

KEY TYPES
  IngredientDetail  — product_id, name, quantity_g, cost_eur,
                      calories_kcal, allergens: list[str]
  DishDetail        — dish_name, description, ingredients,
                      total_cost_eur, total_calories_kcal,
                      allergens: list[str]   (union across all ingredients)
  TrackSummary      — dishes (dict[day, DishDetail]), weekly totals, allergens
  WeekSummary       — week_start, meat (TrackSummary), vegetarian (TrackSummary)

KEY FUNCTIONS
  build_summary(plan, catalogue) → WeekSummary
  summary_to_dict(summary, week) → dict   (API response, includes description)
  to_markdown(summary) → str


================================================================================
MODULE: engine/fallback.py
================================================================================

PURPOSE
  Builds a valid WeeklyPlan from past saved menus when the LLM is unavailable.

HOW IT WORKS
  Reads all *.json files in data/menus/ (skips current week's file).
  For each track, collects all past Dish objects, picks 5 with distinct
  ingredient sets (uniqueness by frozenset of product_ids, not dish name).
  Raises FallbackError if not enough distinct dishes (e.g. first week ever).

FUNCTIONS
  build_fallback_plan(week_start, menus_dir, exclude_path) → WeeklyPlan
  _load_past_dishes(menus_dir, exclude_path) → dict[track, list[Dish]]
  _pick_distinct(dishes, n) → list[Dish]

PURPOSE
  One-time script that enriches products.csv with a cuisine_tag column.

================================================================================
API: api/main.py  (FastAPI)
================================================================================

Routes all include {year} so menus across calendar years work correctly.

  POST /api/generate/{year}/{week}
    Full pipeline: compose → LLM → fix_budget → validate → export → save.
    If LLM fails: fallback from past weeks.
    If fallback fails: 503.
    Saves result to data/menus/{year}_w{week:02d}.json.
    Returns WeekSummary dict. Includes fallback=true if fallback was used.

  GET  /api/menu/{year}/{week}
    Returns stored WeekSummary. 404 if not yet generated.

  PUT  /api/menu/{year}/{week}/{track}/{day}
    Chef edits one dish. Validates before saving. Returns updated WeekSummary.
    Body: { dish_name: str, ingredients: [{product_id, quantity_g}] }

  GET  /api/catalogue
    Returns all available products (used by the UI ingredient search).

  GET  /
    Serves index.html.

  /static/*
    Serves static/ directory (CSS, JS).
    TODO: rename static/ → webapp/


================================================================================
EDGE CASES HANDLED
================================================================================

  LLM API error / timeout              → PlannerError → fallback
  No past menus for fallback (week 1)  → FallbackError → 503
  LLM returns wrong JSON structure     → Pydantic error → retry (3x)
  LLM hallucinates product IDs         → _validate_ids() → error fed back → retry
  LLM flags dish as incoherent         → re-compose that slot → retry
  Still flagged after 3 attempts       → accept + log warning (chef edits)
  Groq rate limit 429                  → backoff 2s/5s/10s (separate counter)
  Week cost exceeds €20.10             → fix_budget() swaps priciest ingredient
  Budget swap would duplicate a dish   → skip, try next cheapest alternative
  Composer generates duplicate dish    → seen_sets check, retry up to 20x
  Empty candidate pool for a role      → ComposerError immediately
  Fruits/sweeteners in sauce slot      → excluded from ROLE_GROUPS (data intact)
  Meat in vegetarian track             → validator catches it (generate + Edit)
  Meat track has no meat protein       → validator catches it
  Unknown product_id in saved plan     → validator reports it
  Invalid year/week in URL             → 400 from _week_start()
  Menu not yet generated               → 404, UI shows empty state
  Year boundary in week navigation     → JS changeWeek() rolls year correctly
  Chef saves invalid dish via Edit     → PUT validates before writing → 422


================================================================================
FRONTEND: static/
================================================================================

Three files — no framework, no build step, plain browser JS.

  index.html  — HTML structure + three modals
  style.css   — all visual styling
  app.js      — all behaviour

STATE (app.js)
  currentYear, currentWeek  — ISO year/week currently displayed
  menuData                  — last fetched WeekSummary (or null)
  catalogue                 — all products, loaded once on init

WEEK/YEAR NAVIGATION
  getISOWeek / getWeeksInYear / isoWeekMonday / formatDateRange
  changeWeek(delta) — rolls year on boundary crossing

API CALLS
  loadCatalogue()   — GET /api/catalogue
  loadWeek()        — GET /api/menu/{year}/{week}
  generateWeek()    — POST /api/generate/{year}/{week}
  saveDish()        — PUT /api/menu/{year}/{week}/{track}/{day}

RENDERING
  renderCalendar(data) — 6-column CSS grid: [label][Mon][Tue][Wed][Thu][Fri]
    Day headers show real dates (e.g. "Mon, Jun 23")
  buildDishCard(track, day, dish)
    View mode:  dish name · cost/kcal · allergen pills · Recipe · Edit buttons
    Edit mode:  name input · ingredient rows (qty editable) ·
                + Add ingredient · Save / Cancel

MODALS
  Recipe modal (#details-overlay)
    "Recipe overview" label + description paragraph
    Table: Ingredient | Grams | Allergens
    Footer: allergen pills + total cost + calories

  Add-ingredient modal (#modal-overlay)
    Search + select from full catalogue + quantity input → appends row
    Does not save — chef must click Save on the dish card.

  Global warning banner (#global-warning)
    Shown when fallback plan was used instead of LLM.
